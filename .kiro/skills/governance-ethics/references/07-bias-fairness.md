# Bias & Fairness Checks

## Why this matters here
CALM features touch crew scheduling, leave, FMLA, and FAA-Part-117 rest requirements. Even a deterministic rule engine
can produce disparate-impact outcomes if the input data is skewed. Code generated by AI must not amplify those skews.

## Checks the agents apply

### Feature Builder
When decomposing a feature involving eligibility, exclusion, or prioritisation:
- Make protected-attribute use explicit. Flag any acceptance criterion that branches on a protected attribute
  (age, gender, race, disability, religion) and surface it for human decision.
- Confirm the rule is grounded in policy or regulation (FAA, FMLA) — never invent eligibility logic.
- Out-of-scope: discretionary scoring or ranking that lacks a documented rule.

### Developer
- Calculations on schedule data must be deterministic and auditable.
- No probabilistic models on individual crew scheduling outcomes without explicit governance review.
- Float arithmetic on hours / minutes uses integer minutes internally; format only at the boundary.
- Date-only strings parsed via the `parseLocalDate` pattern (UI rule); avoid timezone-induced day rollback.

### Test Engineer
- Edge cases include the boundary of every protected-attribute path (where applicable per spec, never generated):
  - Maximum allowed duty period
  - Minimum required rest
  - Pre/post DST transitions
  - Edge of FAA-Part-117 lookback windows
- Test data factories must produce a balanced sample across affected categories so coverage tests do not hide
  a single-bucket bug.

### Code Reviewer
- Flag any branch that uses a protected attribute or proxy without an explicit rule citation.
- Flag any "magic threshold" without policy reference.

## Reporting
The sprint scorecard includes a fairness section: count of decisions involving a protected attribute, decisions
referenced to policy, decisions deferred to human, and any open issues from the adversarial-test pass.
